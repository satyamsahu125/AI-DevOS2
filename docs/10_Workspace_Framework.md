# AI DevOS — Workspace Framework

Version: 1.0

---

# 1. Purpose

The Workspace Framework is responsible for managing the project's workspace metadata.

Unlike source-code parsers or Retrieval-Augmented Generation (RAG) systems, the Workspace Framework does **not** continuously scan the filesystem or load entire files into context.

Instead, it maintains an indexed representation of the workspace that allows other components to quickly locate relevant project assets.

---

# 2. Design Philosophy

The Workspace Framework follows one fundamental rule.

> Index metadata.
>
> Never own source code.

Source code always remains inside the project workspace.

The framework stores only metadata that helps the Context Framework identify relevant files.

---

# 3. Responsibilities

The Workspace Framework is responsible for:

- indexing project files
- tracking modules
- maintaining checksums
- detecting file changes
- exposing workspace metadata
- supporting ContextBuilder

The Workspace Framework never:

- edits source code
- executes code
- stores business memory
- calls LLMs
- reviews artifacts

---

# 5. Workspace Components

The Workspace subsystem consists of:

```
WorkspaceRegistry

WorkspaceIndex

WorkspaceParser

WorkspaceSelector
```

Each component owns exactly one responsibility.

---

# 6. Workspace Registry

The Workspace Registry is the central catalog of the project.

Responsibilities

- register files
- lookup files
- update metadata
- expose indexed information

The registry never reads file contents.

---

# 7. Workspace Index

Each indexed file is represented by a WorkspaceIndex.

Typical fields

- path
- module
- file type
- checksum
- last modified
- metadata

The index contains metadata only.

---

# 8. Workspace Lifecycle

When a file is created

```
File Created

↓

Parser

↓

WorkspaceIndex

↓

Registry
```

When a file changes

```
File Modified

↓

Checksum Updated

↓

Metadata Updated

↓

Registry
```

The source file itself remains unchanged.

---

# 9. Workspace Parser

The WorkspaceParser extracts metadata from files.

Current implementation

- checksum generation

Future implementations

- AST parsing
- import graph extraction
- class indexing
- function indexing
- symbol tables

The parser never communicates with AI agents.

---

# 10. Workspace Selector

The Workspace Selector provides workspace information to the Context Builder.

Flow

```
Workspace Registry

↓

Workspace Selector

↓

Context Builder

↓

Agent Context
```

The selector chooses only the metadata required by the current workflow stage.

---

# 11. Metadata Model

Each WorkspaceIndex contains metadata describing the file.

Examples

- module ownership
- language
- generated by
- artifact type
- checksum
- timestamps

Metadata is intentionally lightweight.

---

# 12. Why Metadata Instead of Source Code?

Scanning the entire repository for every execution is expensive.

Advantages of indexing metadata:

- faster context construction
- deterministic execution
- lower memory usage
- provider independence
- scalable repositories

Source code is retrieved only when explicitly required.

---

# 13. Relationship with Context Framework

The Context Framework never scans the workspace.

Instead

```
Workspace Registry

↓

Workspace Selector

↓

Agent Context
```

This keeps ContextBuilder independent from filesystem operations.

---

# 14. Relationship with Memory Framework

Workspace metadata is **not memory**.

Memory answers:

"What has the project learned?"

Workspace answers:

"What files currently exist?"

The two systems remain independent.

---

# 15. Relationship with Execution Framework

The Execution Framework never interacts directly with the workspace.

Instead

```
Execution

↓

Context Builder

↓

Workspace Selector

↓

Workspace Registry
```

This preserves execution isolation.

---

# 16. Checksum Tracking

Each file maintains a checksum.

Purpose

- detect modifications
- invalidate caches
- identify outdated indexes
- support future incremental context updates

Current implementation uses SHA-256.

# 20. Runtime Guarantees

The Workspace Framework guarantees

✓ Source code remains on disk

✓ Registry owns metadata only

✓ ContextBuilder never scans files

✓ Workspace metadata is deterministic

✓ File indexing remains centralized

✓ Workspace and Memory remain independent

---

# 21. Design Decisions

The Workspace Framework intentionally separates:

Filesystem

↓

Workspace Registry

↓

Workspace Selector

↓

Context Builder

↓

Agent

This avoids repeated filesystem traversal and enables predictable context construction.

---

# 22. Summary

The Workspace Framework provides an indexed view of the project without owning or duplicating source code.

By maintaining lightweight metadata through the Workspace Registry and exposing it through the Workspace Selector, AI DevOS can efficiently construct execution contexts while remaining scalable, deterministic, and independent of repository size.
